# 图书销售管理系统项目文档（中期实验）

## 1. 项目概述

### 1.1 项目名称
图书销售管理系统的设计与实现

### 1.2 项目背景
某书城需要一套管理系统，对图书的进货、销售、库存和财务进行统一管理，提升日常运营效率，降低人工统计和手工对账带来的错误风险。

### 1.3 项目目标
- 实现书城核心业务流程的信息化管理。
- 建立结构合理、可扩展的数据库，保证数据一致性。
- 支持多角色权限控制，保障系统安全性。
- 满足实验基础功能要求，并预留扩展能力。

---

## 2. 角色与权限设计

系统用户分为两类：

### 2.1 超级管理员
- 可执行普通管理员的全部操作。
- 可创建普通管理员账号。
- 可查看所有用户资料。

### 2.2 普通管理员
- 可管理图书进货、销售、库存等业务信息。
- 仅可查询和修改自己的用户信息。

### 2.3 登录与安全要求
- 所有功能均需登录后才能使用。
- 用户密码不得明文存储，需加密后写入数据库（实验要求建议使用 MD5；实际生产建议使用更安全算法如 bcrypt/argon2）。
- 用户基础信息包含：用户名、密码、真实姓名、工号、性别、年龄等。

---

## 3. 功能需求说明

## 3.1 用户管理
- 系统预置超级管理员账号（初始化时已存在）。
- 超级管理员创建普通管理员账号。
- 用户个人信息查询与修改。

## 3.2 库存书籍管理
- 维护库存书籍基础信息：
  - 书籍编号
  - ISBN
  - 书名
  - 作者
  - 出版社
  - 零售价格
  - 当前库存数量

## 3.3 书籍查询
- 支持按以下条件查询：
  - 书籍编号
  - ISBN
  - 书名
  - 作者
  - 出版社

## 3.4 图书信息修改
- 可修改书名、作者、出版社、零售价格等可维护字段。

## 3.5 图书进货管理
- 对已存在图书：直接加入进货清单。
- 对新图书：需录入 ISBN、书名、作者、出版社等信息后加入进货清单。
- 每条进货记录需包含进货价格、进货数量。
- 新增进货记录时，初始状态为“未付款”。

## 3.6 进货付款
- 查询处于进货中的图书记录。
- 完成付款后，进货状态变更为“已付款”。

## 3.7 图书退货
- 仅允许对“未付款”进货记录执行退货。
- 退货后状态变更为“已退货”。

## 3.8 添加新书入库
- 对“已付款”且已到货的进货记录执行入库。
- 入库时需设置/确认零售价格。
- 入库后更新库存数量。

## 3.9 图书销售
- 按零售价格售卖图书。
- 销售成功后库存数量减少。

## 3.10 财务管理
- 进货付款时记一条“支出”账单。
- 图书销售时记一条“收入”账单。
- 支持查看某时间区间内账单（收入/支出）。

---

## 4. 业务流程（建议实现）

### 4.1 进货到入库流程
1. 创建进货单（未付款）  
2. 付款（状态改为已付款，产生财务支出）  
3. 到货后入库（增加库存）  

### 4.2 销售流程
1. 查询图书库存  
2. 创建销售记录  
3. 扣减库存  
4. 生成财务收入账单  

### 4.3 退货流程
1. 选择未付款进货记录  
2. 执行退货  
3. 状态改为已退货（不入库、不产生付款）

## 5. 数据库设计（参考）

为满足实验评分中“数据库设计（30%）”要求，建议至少设计以下核心表：

### 5.1 `users`（用户表）
- `id`（PK）
- `username`（唯一）
- `password_hash`
- `real_name`
- `employee_no`（唯一）
- `gender`
- `age`
- `role`（super_admin / admin）
- `created_at`、`updated_at`

### 5.2 `books`（图书基础信息表）
- `id`（PK）
- `isbn`（唯一）
- `title`
- `author`
- `publisher`
- `retail_price`
- `stock_qty`
- `created_at`、`updated_at`

### 5.3 `purchase_orders`（进货单主表）
- `id`（PK）
- `status`（unpaid / paid / returned / stocked）
- `operator_id`（FK -> users.id）
- `total_amount`
- `created_at`、`updated_at`

### 5.4 `purchase_order_items`（进货单明细）
- `id`（PK）
- `purchase_order_id`（FK）
- `book_id`（FK，可为空用于新书场景）
- `isbn`
- `title`
- `author`
- `publisher`
- `purchase_price`
- `quantity`
- `subtotal`

### 5.5 `sales_orders`（销售单）
- `id`（PK）
- `book_id`（FK）
- `sale_price`
- `quantity`
- `total_amount`
- `operator_id`（FK -> users.id）
- `created_at`

### 5.6 `finance_records`（财务流水）
- `id`（PK）
- `type`（income / expense）
- `amount`
- `related_type`（purchase / sale）
- `related_id`（关联业务单据ID）
- `remark`
- `operator_id`（FK -> users.id）
- `created_at`

### 5.7 关键约束建议
- 用户名、工号、ISBN 建立唯一索引。
- 金额字段使用 `decimal` 类型，避免浮点误差。
- 所有库存变更、付款、销售建议放在事务中处理。
- 对高频查询字段（isbn、title、author、created_at）建立索引。

## 6. 非功能要求与实现建议

### 6.1 技术选型
实验对语言、框架、数据库不限，可选：
- 桌面端：C++/MFC 等
- Web：ASP.NET / JSP / PHP / Java / Python / Node.js 等
- 命令行程序也可接受

### 6.2 UI 要求
- 本实验不强调界面美观，强调操作简便、流程完整、功能正确。

### 6.3 测试建议
- 使用自建测试数据验证所有核心业务流程。
- 重点覆盖：
  - 权限控制
  - 库存增减正确性
  - 财务流水与业务单据一致性
  - 异常场景（库存不足、重复ISBN、未登录访问）

## 11. 结语

本项目以图书销售管理为核心场景，通过“用户权限 + 库存管理 + 进销存流程 + 财务流水”形成完整业务闭环。实现时应优先保证数据库设计正确、核心流程可用、测试与报告规范，以满足课程实验要求并具备一定工程实践价值。
